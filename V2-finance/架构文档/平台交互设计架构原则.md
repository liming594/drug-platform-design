# 平台交互设计架构原则

> 版本：v1.0 | 日期：2026-06-20
> 来源：财务模块设计讨论中提炼的平台级架构原则
> 适用范围：平台所有涉及审批的业务模块（财务/用章/物资/合同/培训等）

---

## 一、核心原则：审批中心不拥有数据，只展示切片

### 1.1 问题背景

老OA的混乱根源之一：审批和业务数据混在一起。每次找一个东西，不知道它是走了借款还是报销还是合同用款，选择很多、路径不同、数据分散。新平台必须从架构层面解决这个问题。

### 1.2 核心分离

| 职责 | 归属 | 说明 |
|------|------|------|
| **审批模板/模型** | 审批中心 | 几级审批、审批人条件、审批规则，统一在审批中心配置 |
| **审批流程调度** | 审批中心 | 发起→流转→通过/驳回的状态机引擎 |
| **业务数据** | 各业务模块 | 申办方合同、支出流水、用章记录等主数据存在各自对象表 |
| **审批内容** | 各业务模块 | 提交的表单内容、附件、金额等，都是业务对象的一部分 |

**一句话总结**：审批中心管流程模板和调度，不碰业务数据；业务模块管数据，不碰审批引擎。两边通过接口对接。

### 1.3 交互流程

```
员工在审批中心入口提交
  ↓
业务模块创建业务数据（存在业务模块自己的表里）
  ↓
业务模块调用审批中心模板，发起审批流程
  ↓
审批中心负责流程流转（几级审批、谁审、怎么流转）
  ↓
审批通过/驳回 → 审批中心回调业务模块
  ↓
业务模块更新自己的状态（APPROVED/REJECTED → PAID/CLOSED等）
```

### 1.4 审批中心的"切片"本质

审批中心不存任何业务数据。它呈现的内容，是以当前用户视角从各业务模块查出来的投影：

- **员工视角**：进入审批中心 → 财务审批历史 → 看到的是自己提交的财务记录切片（从fin_payment_records WHERE created_by=当前用户）
- **管理层视角**：看到全部切片（从各业务模块全量查询）
- **审批人视角**：看到待自己审批的切片（从审批中心查待办，关联到业务模块的数据）

审批中心里的每个入口（财务/用章/物资/培训等），映射到对应业务模块的主对象。入口本身不持有数据，只是一个人视角的快捷通道。

## 二、三层入口架构

### 2.1 架构概览

所有涉及审批的业务模块，统一采用三层入口：

| 层 | 入口位置 | 面向角色 | 能做什么 | 不能做什么 |
|---|---|---|---|---|
| **提交层** | 审批中心 → 对应入口 | 所有员工 | 选类型→填表单→附材料→提交 | — |
| **监管层A** | 项目运营/业务模块 → 对应Tab | 业务负责人（PM等） | 查看本项目/本业务域历史记录、统计、筛选、管理已有单据 | 不能创建新记录 |
| **监管层B** | 综合管理 → 对应管理模块 | 管理人员/财务等 | 全部记录的增删改查、审批、确认、结算 | — |

### 2.2 三层职责详解

#### 提交层（审批中心）
- **定位**：员工最快捷的操作入口
- **设计要求**：信息完整全面 + 界面简洁干净 + 操作快捷舒服
- **表单设计**：按类型动态展示字段，不展示无关项；选择业务来源后自动带出关联信息
- **视角**：员工只看到自己提交的历史记录

#### 监管层A（项目运营/业务模块内）
- **定位**：业务负责人的监管视角
- **能力**：查看、筛选、统计本项目/本业务域的历史记录，可做管理操作（改备注、标记关注等）
- **限制**：不提供"新建"入口——创建统一走审批中心
- **视角**：PM看到本项目所有相关记录

#### 监管层B（综合管理）
- **定位**：管理人员的全局视角
- **能力**：全权限——增删改查、审批、确认、结算、补票等
- **视角**：财务看到全平台所有记录，按需筛选

### 2.3 为什么这样设计

1. **创建只有一条路**：员工不用纠结去哪提交，统一走审批中心，避免碎片化
2. **监管有多条路**：PM在项目运营里看本项目，财务在综合管理里看全平台，各自视角不同但看的是同一份数据
3. **项目/业务模块是显示层**：能看到、能筛、能统计，但不提供创建入口
4. **同一条记录双向可见**：PM和财务看到的是同一条数据，都能做各自权限内的管理操作

### 2.4 项目交接与历史可见性

**痛点**：PM在审批中心只能看到自己提交/审批的记录。项目交接后，前任PM经手的费用、用章、物资等记录，新PM在审批中心看不到——但他承接了项目，需要看到全部历史。

**解决方案**：项目运营入口是PM看全历史的唯一正确路径。

| 场景 | 审批中心能看到什么 | 项目运营能看到什么 |
|------|-------------------|-------------------|
| 在任PM自己提的 | ✅ 自己的切片 | ✅ 本项目全部 |
| 前任PM提的（已交接） | ❌ 看不到 | ✅ 本项目全部 |
| 离职员工提的 | ❌ 看不到 | ✅ 本项目全部 |
| 其他部门提的关联费用 | ❌ 看不到 | ✅ 本项目全部 |

**关键**：项目运营以project_code为锚点查全部记录，不受"是谁提的"限制。审批中心以user_id为锚点查个人切片，受限于提交人。

### 2.5 单据转移与人员变动管理

项目交接或人员离职时，各层面的管理者可以操作单据转移：

| 转移场景 | 操作入口 | 操作能力 |
|----------|----------|----------|
| 项目交接 | 项目运营→财务Tab | 将离职人员的历史费用单据转给接手人 |
| 人员离职 | 综合管理→财务管理 | 将离职人员所有待审批/待处理单据批量转移 |
| 用章记录转移 | 综合管理→用章管理 | 将离职人员的用章记录转给接手人 |
| 费用审批权转移 | 综合管理→对应模块 | 将A的审批权转给B（如A离职，B接手） |

**转移内容**：
- 单据归属人（created_by / applicant）
- 待审批任务的审批人
- 借款余额的还款责任人
- 供应商付款的经办人

**设计原则**：转移操作记录日志，谁转的、转给谁、转了哪些单据、什么时间，全部留痕。

### 2.6 多层面管理权限梯度

每个层面都能管理，区别在于权限范围大小：

```
权限范围由小到大：

员工个人（审批中心切片）
  → 只看/管自己提交的
  
项目经理（项目运营 → 各业务Tab）
  → 看/管本项目所有记录（含前任和离职人员的）
  → 可做单据转移（项目层面）
  
中心负责人（中心管理 → 各业务Tab）
  → 看/管本中心所有项目的记录
  → 可做单据转移（中心层面）
  
财务/行政部门（综合管理 → 各管理模块）
  → 看/管全平台所有记录
  → 全权限：增删改查、审批、确认、结算、补票
  → 可做单据转移（全局层面）
  
admin
  → 全部权限
```

**核心思路**：不是"只有财务能管"，而是每个层面都能管自己范围内的，只是权限大小不同。项目层面能转移单据、中心层面能转移单据、综合管理层面也能——只不过范围递增。

## 三、适用范围

### 3.1 已确认适用的业务模块

| 业务模块 | 提交层入口 | 监管层A | 监管层B | 主数据存储 |
|----------|-----------|---------|---------|-----------|
| **财务管理** | 审批中心→财务/借款/报销 | 项目运营→财务Tab | 综合管理→财务管理 | finance.db |
| **用章管理** | 审批中心→用章申请 | 项目运营→用章记录 | 综合管理→用章管理 | seal.db（16号线） |
| **物资管理** | 审批中心→物资申请 | 项目运营→物资记录 | 综合管理→物资管理 | material.db（14号线） |
| **培训管理** | 审批中心→培训申请 | 项目运营→培训记录 | 综合管理→培训管理 | training.db（13号线） |
| **会议管理** | 审批中心→会议申请 | 项目运营→会议记录 | 综合管理→会议管理 | meeting.db（15号线） |
| **合同管理** | 审批中心→合同审批 | 项目运营→合同记录 | 综合管理→合同管理 | contract.db |

### 3.2 通用规则

每个业务模块遵循以下规则：

1. **主数据归各模块**：业务记录存在各模块自己的独立库，审批中心不复制不存储
2. **模板归审批中心**：审批级数、审批人条件、流转规则在审批中心统一配置
3. **提交走审批中心**：员工创建新记录统一从审批中心入口进入
4. **监管走业务模块**：PM和财务/管理人员在各自的管理入口查看和管理
5. **接口对接**：业务模块通过 `POST /api/approval/instances/submit-from-module` 发起审批，审批中心通过回调通知结果

## 四、以财务模块为首个落地案例

### 4.1 财务模块5线架构

| 线 | 模块 | 主数据表 | 业务关联 |
|---|---|---|---|
| **收入线** | 收款管理 | fin_collections / fin_invoices | 申办方合同打款 |
| **支出线** | 统一支出流水 | fin_payment_records | 报销/借款/供应商付款 |
| **票据视图** | 发票提取视图 | （从收支表提取） | 被动提取不录入 |
| **汇总线** | 结算+盈亏 | fin_settlements / fin_project_pl | 汇总收支数据 |
| **审计线** | 审计分桶 | （从各表提取） | 异常检测 |

### 4.2 财务模块三层入口映射

| 层 | 入口 | 看到什么 | 能做什么 |
|---|---|---|---|
| 提交层 | 审批中心→财务/借款/报销 | 自己提交的历史 | 选类型→选预算→填金额→附票据→提交 |
| 监管层A | 项目运营→财务Tab | 本项目所有收支记录 | 查看、筛选、统计、管理已有单据（无新建） |
| 监管层B | 综合管理→财务管理 | 全平台所有收支记录 | 增删改查、审批、确认、补票、结算 |

### 4.3 财务模块提交层表单

**通用头部**：支出类型切换（报销/借款/供应商付款），切换后下方字段动态变化

**日常报销（EXPENSE）**：
- 项目选择（可选）→ 选了走项目预算 / 不选走部门预算
- 费用类型（差旅/办公/试验/餐饮/其他）
- 金额 + 事由说明
- 票据信息（发票号/不含税金额/税率/附件）— 有票才报销

**员工借款（LOAN）**：
- 部门选择
- 借款金额 + 事由
- 预计还款日期

**供应商付款（VENDOR_PAYMENT）**：
- 项目选择（必选）
- 业务来源（SMO合同/合作商合同/中心合同）→ 选合同→自动带出付款节点
- 付款节点 → 自动带出计划金额
- 实际付款金额 + 事由
- 有发票→填发票信息 / 无发票→标记待补票

### 4.4 支出类型核心规则

| 规则 | 说明 |
|------|------|
| **不合并** | 报销就是报销，供应商付款就是供应商付款，各走各的流程 |
| **报销双轨预算** | 选了项目→项目预算（P&L可见）；没选项目→部门预算 |
| **报销必须有票** | has_invoice=1, invoice_status=ATTACHED |
| **供应商付款可先付后票** | 付款时无发票→PENDING_INVOICE→还票提醒→补票后RECONCILED→CLOSED |
| **借款不计入成本** | LOAN类型不计入P&L的total_cost |
| **供应商付款关联业务源头** | source_module + source_contract_id + source_payment_node_id |

### 4.5 票据视图：提取而非录入

- 销项发票：从fin_invoices提取（收款后开票，跟收款走）
- 进项发票：从fin_payment_records发票字段提取（跟着支出走）
- 待补票提醒：从fin_payment_records WHERE invoice_status=PENDING_INVOICE
- 不做独立发票录入模块，避免"两头做"的割裂

## 五、技术实现要点

### 5.1 审批中心接口约定

```
业务模块 → 审批中心（发起审批）：
POST /api/approval/instances/submit-from-module
{
  "module": "finance",           // 业务模块标识
  "business_type": "expense",    // 业务类型
  "business_id": "PAY-20260620-001",  // 业务记录ID（在业务模块自己的表中）
  "flow_code": "finance_expense",     // 审批模板编码（在审批中心配置）
  "remark": "XX项目差旅报销",
  "extra_data": { "amount": 5000, "project_code": "SINO202505001" }
}

审批中心 → 业务模块（回调通知）：
回调业务模块的 webhook：
- approved → 业务模块执行后续操作（PAID等）
- rejected → 业务模块回退状态 + 记录驳回原因
```

### 5.2 切片查询约定

员工在审批中心查看历史时，审批中心代理查询各业务模块：

```
审批中心 → 业务模块（查询用户切片）：
GET /api/finance/payments?created_by={user_id}&page=1&size=20
→ 返回该用户提交的支出流水记录
```

### 5.3 模板配置

审批模板在审批中心统一配置，按金额/类型等条件路由：

| 模板编码 | 触发条件 | 审批流 |
|----------|----------|--------|
| finance_expense_le5k | 报销 ≤5,000 | 部门经理 → 财务确认 |
| finance_expense_5k_50k | 报销 5K~50K | 部门经理 → 财务总监 |
| finance_expense_gt50k | 报销 >50K | 部门经理 → 财务总监 → CFO |
| finance_loan | 借款 | 部门经理 → 财务总监 |
| finance_vendor_payment | 供应商付款 | 部门经理 → 财务总监（>50K加CFO） |
| finance_collection_confirm | 收款确认 | 财务专员 → 财务总监 |
| finance_invoice_issue | 开票申请 | 财务专员 → 财务总监 |
| finance_settlement_confirm | 结算确认 | 财务总监 |
| seal_apply | 用章申请 | 部门经理 → 行政确认 |
| material_request | 物资申请 | 部门经理 → 行政确认 |

## 六、与现有代码的关系

### 6.1 现有审批中心

当前平台的审批中心（基于BPM流程表）已具备基础能力，需扩展：
- 新增各业务模块的入口配置
- 新增切片查询代理能力（按用户ID从各业务模块查询）
- 模板配置界面扩展（支持金额分级路由）

### 6.2 现有业务模块改造

各业务模块需新增：
- 提交接口：接收审批中心转发的提交请求，创建业务数据，调用审批中心发起流程
- 回调接口：接收审批中心的审批结果回调
- 切片查询接口：支持按created_by查询（供审批中心代理调用）

### 6.3 数据归属对照

| 数据 | 存在哪儿 | 不存在哪儿 |
|------|----------|-----------|
| 支出流水（报销/借款/供应商付款） | finance.db → fin_payment_records | 审批中心 |
| 收款记录 | finance.db → fin_collections | 审批中心 |
| 用章记录 | seal.db → seal_records | 审批中心 |
| 审批流程实例 | 审批中心（BPM表） | 各业务模块 |
| 审批模板配置 | 审批中心 | 各业务模块 |
| 审批流转日志 | 审批中心 | 各业务模块 |
