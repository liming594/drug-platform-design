# 项目中心层面费用管理盘点

> 盘点日期：2026-06-20
> 盘点目的：确认项目运营和中心管理两个层面的费用/预算代码能否通过小幅改动对接三层入口架构

---

## 一、概述

根据《平台交互设计架构原则》定义的三层入口架构：

| 层 | 入口位置 | 面向角色 | 核心限制 |
|---|---|---|---|
| **提交层** | 审批中心 | 所有员工 | 创建新记录的唯一入口 |
| **监管层A** | 项目运营 → 财务/预算Tab | PM等业务负责人 | 只读+管理，**不能新建** |
| **监管层B** | 综合管理 → 财务/预算管理 | 财务/管理人员 | 全权限 |

本次盘点覆盖5个前端JS文件，分析其在费用/预算管理上的现有能力与三层架构的适配差距，评估改动量。

## 三、逐文件分析

### 3.1 pm-finance-module.js — 项目运营费用切片

**1. 当前功能**

项目运营层面的财务费用只读切片视图。以当前项目（可选缩放到中心）为锚点，展示费用KPI和明细表。

- 页面结构：标题栏 → KPI卡片（应收/已收/应付/已付/发票缺口）→ 费用明细表格
- 内置审批流切片弹窗（可查看单条费用记录在审批中心的审批实例详情）

**2. 费用相关能力**

- ✅ 费用KPI查看：应收、已收、应付/费用、已付、发票缺口
- ✅ 费用明细列表：单号、类型、对象/人员、合同/来源、金额、状态、下一步
- ✅ 审批流切片查看：从审批中心拉取审批实例，展示流程节点、处理日志
- ✅ 快捷跳转：跳预算校验（`pm_budget`）、跳合同依据（`pm_contracts`）

**3. 数据来源**

- 主数据API：`GET /api/projects/{projectId}/finance/summary`（或 `.../sites/{siteId}/finance/summary` 缩放到中心）
- 审批流API：`GET /api/approval/instances/query?module=finance&keyword={businessId}` → `GET /api/approval/instances/{id}`
- Token机制：`ensureLocalMirrorToken` + `getAccessToken`/localStorage多键兜底

**4. 操作能力**

| 操作 | 支持 | 说明 |
|------|------|------|
| 查看 | ✅ | KPI + 明细表 |
| 新建 | ❌ | 无新建入口 |
| 编辑 | ❌ | 无编辑入口 |
| 删除 | ❌ | 无删除入口 |
| 审批 | ❌ | 只读查看审批流，不做审批操作 |
| 导出 | ❌ | 无导出功能 |
| 审批流查看 | ✅ | 弹窗展示审批实例详情 |

**5. 与三层入口的差距**

| 检查项 | 状态 | 说明 |
|--------|------|------|
| 是否有"新建"入口 | ✅ 无 | 已符合监管层A"不能新建"要求 |
| 是否有审批功能 | ✅ 无 | 只读查看审批流，不做审批操作 |
| 数据是否按项目维度查询 | ✅ 是 | 以projectId为锚点，支持缩放到siteId（中心维度） |
| 是否支持单据转移 | ❌ 无 | 缺少单据转移功能（项目交接场景） |
| 是否有全量数据越权 | ✅ 无 | 只查本项目切片 |

**6. 适配建议**

- **保留**：全部现有功能（只读展示 + 审批流切片查看 + 快捷跳转）
- **新增**：单据转移功能（PM在项目交接时转移费用单据归属人）
- **改动量**：小 — 仅需在操作栏新增"转移"按钮和对应弹窗，数据展示部分无需改动

### 3.2 pm-budget-module.js — 项目运营预算及执行

**1. 当前功能**

项目运营层面的预算管理模块，展示项目预算明细、执行消耗、风险预警、变更记录，并提供预算编制和调整入口。

- 页面结构：项目信息卡片 → 预算KPI（批准/已执行/剩余/人工/风险）→ 预算明细表 → 待处理风险卡片 + 变更记录 → 执行监控 + 人工成本 → 执行消耗流水表（支持搜索/筛选/导出）
- 弹窗1：**编制项目预算**（按8项模板目录填写金额 → 提交审批）
- 弹窗2：**申请预算调整**（追加/调减/科目结构调整 → 提交审批）
- 预算模板：8项（中心费用、招募费用、中心实验室、SMO/CRC、CRA监查、数据/统计、质量/文档、物资物流）

**2. 费用相关能力**

- ✅ 预算KPI查看：批准预算、已执行、剩余额度、人工成本、待处理风险
- ✅ 预算明细表：类别/子项、责任口径、预算、已执行、剩余、执行率、来源
- ✅ 执行消耗流水：记录编号、日期、来源、类别、人员/对象、中心、金额、关联单据、状态
- ✅ 风险预警：执行率≥65%标黄、≥85%标红
- ✅ 预算变更记录
- ✅ 执行监控条形图
- ✅ CSV导出
- ⚠️ 预算编制（新建草稿）
- ⚠️ 预算调整申请

**3. 数据来源**

- 主数据API（优先）：
  - `GET /api/projects/{code}/budget/items` — 预算科目
  - `GET /api/projects/{code}/budget/cost-records` — 成本流水
  - `GET /api/projects/{code}/budget/summary` — 汇总
- Fallback API：`GET /api/budgets?tab=budget` — 从预算主台账按项目过滤
- 本地缓存：`localStorage['pm_{projectId}_budget']`
- 保存后端：`pmSaveToBackend('budget', projectId, data)`
- 数据来源标记：`source: 'PROJECT_BUDGET_SLICE_API'` 或 `source: 'BUDGET_MASTER_API'`

**4. 操作能力**

| 操作 | 支持 | 说明 |
|------|------|------|
| 查看 | ✅ | 预算明细+执行流水+KPI+风险 |
| 新建 | ⚠️ 有 | "编制项目预算"按钮 → 弹窗 → 生成草稿提交审批 |
| 编辑 | ❌ | 无直接编辑（但"申请调整"实质是修改预算） |
| 删除 | ❌ | 无删除入口 |
| 审批 | ❌ | 无审批操作（提交后只toast提示） |
| 导出 | ✅ | CSV导出执行消耗流水 |
| 刷新 | ✅ | 手动刷新预算API数据 |

**5. 与三层入口的差距**

| 检查项 | 状态 | 说明 |
|--------|------|------|
| 是否有"新建"入口 | ⚠️ 有 | "编制项目预算"按钮 — **需移除** |
| 是否有审批功能 | ✅ 无 | 无审批操作 |
| 数据是否按项目维度查询 | ✅ 是 | 以projectCode为锚点查询 |
| 是否支持单据转移 | ❌ 无 | 缺少单据转移功能 |
| "申请调整"是否合规 | ⚠️ 需评估 | 调整本质是变更请求，按新架构应走审批中心提交 |

**6. 适配建议**

- **移除**：
  - "编制项目预算"按钮和弹窗（`pmShowProjectBudgetTemplateModal` / `pmSubmitProjectBudgetDraft`）
  - "申请调整"按钮和弹窗（`pmShowProjectBudgetAdjustModal` / `pmSubmitProjectBudgetAdjust`）
  - 空数据页面的"编制项目预算"引导按钮
- **保留**：
  - 全部只读展示功能（KPI、明细表、风险卡片、变更记录、执行监控、消耗流水、导出）
  - 刷新功能
- **新增**：
  - 单据转移功能（预算单据归属人转移）
- **改动量**：小 — 主要移除2个按钮+2个弹窗函数调用入口，展示逻辑不变

### 3.3 ops-budget.js — 综合管理预算工作台

**1. 当前功能**

综合管理层面的预算工作台，面向财务/管理层，覆盖全平台预算的增删改查、审批流集成、项目预算卡片总览。

- 页面结构：标题栏（含"新增预算"按钮）→ 筛选栏（年度/类型/状态/搜索）→ 导入状态提示 → KPI卡片 → 项目预算总览卡片 → 预算监管入口
- 弹窗1：**新增预算**（选类型/年度/项目或部门 → 按模板填写金额 → 生成草稿 → 审批预览）
- 弹窗2：**编辑预算**（修改预算金额、已用金额、状态等）
- 弹窗3：**预算调整**（追加/调减/科目结构调整 → 提交审批）
- 弹窗4：**预算监管**（审批流查看、调整操作）
- 预算模板：8项项目模板 + 4项部门模板
- 项目预算总览：按项目归并展示卡片（预算/已执行/余额/执行率/预警状态）
- 项目预算卡片可跳转至项目运营的预算模块

**2. 费用相关能力**

- ✅ 预算KPI：预算总额、已执行、余额、接近超支、待审批
- ✅ 项目预算卡片总览（一项目一卡）
- ✅ 预算监管入口（审批流查看+调整）
- ✅ 新增/编辑/删除预算
- ✅ 预算调整申请
- ✅ 导入状态检查（检测测试数据、未导入项目预算等）
- ✅ 审批流集成（`opsApprovalFlowButton` / `opsApprovalFormFlowPreview`）

**3. 数据来源**

- API：`GET/POST/PUT/DELETE /api/budgets`，`GET /api/budgets/import-status`
- 审批流：通过 `opsApprovalFlowButton` 和 `opsApprovalOpenFlowSlice` 集成
- Token机制：与pm-budget一致的`ensureLocalMirrorToken` + 多键兜底

**4. 操作能力**

| 操作 | 支持 | 说明 |
|------|------|------|
| 查看 | ✅ | 全平台预算总览+明细 |
| 新建 | ✅ | 新增预算编制弹窗 |
| 编辑 | ✅ | 编辑预算弹窗 |
| 删除 | ✅ | 删除预算记录 |
| 审批 | ✅ | 审批流查看+集成 |
| 调整 | ✅ | 预算调整申请 |
| 导出 | ❌ | 无导出功能 |

**5. 与三层入口的差距**

| 检查项 | 状态 | 说明 |
|--------|------|------|
| 是否有"新建"入口 | ✅ 有 | 监管层B应有此能力，**符合** |
| 是否有审批功能 | ✅ 有 | 监管层B应有此能力，**符合** |
| 数据是否全平台范围 | ✅ 是 | 查询全平台预算，**符合** |

- **改动量**：小 — 主要是提交逻辑对接审批中心接口，UI层基本不变

### 3.4 ops-core.js — 综合管理基础架构

**1. 当前功能**（仅分析前200行结构）

综合管理的基础架构层，定义所有模块列表、元数据和审批流基础函数。

- `opsViewModules` 数组：列出全部22个子模块，含 `ops_budget`（预算及执行）、`ops_finance`（费用管理）、`ops_approval`（审批中心）
- `opsSubModuleMeta` 对象：每个模块的名称、描述、图标、颜色定义
- `opsOverviewGroups`：6大分组（客户与商务、合作资源主数据、组织人力与培训、财务合同与章证、质量合规与文档、行政协同）
- 审批流基础函数：`approvalStatusText`、`approvalInferFlowCode`、`approvalFlowButtonAttrs`、`approvalMetaFromElement` 等
  - 支持推断审批流编码（如 `AP-PRJ-PAYMENT`、`AP-DEPT-EXPENSE`）
  - 支持流程解析 `/api/approval/flows/resolve`
  - 支持业务实例查询 `/api/approval/instances/business`

**2. 费用相关**

- `ops_budget` 模块定义：`'预算及执行'` — '公司管理层和财务查看平台级项目预算健康度、预算新增、监管和调整'
- `ops_finance` 模块定义：`'费用管理'` — '应收、应付、报销、付款、发票、借款和实际支出流水'
- 审批流编码推断：`finance` → `AP-PRJ-PAYMENT` 或 `AP-DEPT-EXPENSE`；`budget` → `AP-PRJ-BUDGET` 或 `AP-DEPT-BUDGET`

**3. 与三层入口的差距**

| 检查项 | 状态 | 说明 |
|--------|------|------|
| 模块定义是否完整 | ✅ 是 | 预算和费用模块已定义 |
| 审批流推断是否支持 | ✅ 是 | 已有按module+businessType推断flowCode的逻辑 |
| 分组是否合理 | ✅ 是 | 财务合同与章证分组包含finance模块 |

**4. 适配建议**

- **保留**：全部现有定义
- **可选优化**：更新 `ops_budget` 和 `ops_finance` 的描述文字，明确三层入口定位
- **改动量**：极小 — 仅描述文字更新，无功能改动

## 七、结论

### 能否小幅改动对接三层入口架构？

**可以。**

核心判断依据：

1. **项目运营层面（监管层A）**：
   - `pm-finance-module.js` 已经是完全只读设计，无新建/编辑/删除/审批入口，**零改动即可符合**
   - `pm-budget-module.js` 只需移除2个按钮（编制预算+申请调整）及其弹窗调用入口，展示逻辑完全保留，**改动量约35行代码**
   - 两个文件的数据查询均已按project_code为锚点，符合"项目维度全历史可见"要求

2. **中心管理层面（监管层B）**：
   - `ops-budget.js` 已具备全权限CRUD+审批集成，**功能上已符合**
   - 仅需将提交逻辑对接审批中心标准接口（`POST /api/approval/instances/submit-from-module`），**改动量集中在提交函数**
   - `ops-core.js` 审批流推断逻辑已支持按module+businessType路由flowCode（如 `AP-PRJ-PAYMENT`、`AP-DEPT-EXPENSE`），**与三层架构的审批中心对接已具备基础**

3. **代码复用率**：
   - 现有代码的API封装、Token管理、XSS防护、样式注入等基础设施可完全复用
   - 审批流按钮组件（`opsApprovalFlowButton`）已实现跨模块复用
   - 预算模板（8项项目+4项部门）已定义，无需重新设计

4. **风险点**：
   - `ops-modules.js`（15166行）中的 `ops_finance` 模块尚未深入分析，需后续补充
   - 单据转移功能为新增需求，需设计UI和API，但可参考现有弹窗模式实现

**总改动量评估**：P0项约35行代码删除 + P1项约200行新增代码 = **小改动**，可在1-2个工作日内完成核心适配。
