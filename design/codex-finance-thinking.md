# 财务模块 思路沟通记录

> 记录时间：2026-06-14
> 内容：财务模块从需求提出到设计定稿的完整沟通过程，尽量还原当时的对话和决策

---

## 一、起点：综合管理→财务下面该放什么

**我说的：** 综合管理里面财务分类下面应该有预算及执行模块，这个模块进去之后应该有一个项目预算入口（查询所有项目预算列表的，点击具体项目进去能看到详细信息、历史审批情况等，管理员财务可以进行操作打回监管等），一个部门预算入口（非走项目的其他类费用走部门，同样可以操作），这里面的执行部分数据都是其他部分自动带出来的，预算一旦批了就定了，要改就重新走增改留痕。

**设计回应：** 明白了，这是一个更高层级的预算管理模块——综合管理→财务→预算及执行，比之前的项目级费用追踪多了完整的审批生命周期、部门预算、执行自动汇总和增改留痕。

**核心决策：**
- 预算及执行是综合管理→财务下的独立模块
- 项目预算和部门预算是两个入口
- 执行数据是自动汇总的，不是手动填的
- **预算一旦批了就定了，改就重新走增改留痕**——这句话定下了整个审批流的核心

---

## 二、提请入口在项目中心

**我说的：** 项目预算提请肯定是在项目中心，预算及执行里面提交经过审批后生效。

**设计回应：** 明白，项目预算的提请入口在项目中心，综合管理→财务→预算及执行是全局查看/审批/监控的入口。

**核心决策：**
- **谁干活谁提请**：PM在项目中心提预算，不是在综合管理提
- **管钱的审批监控**：综合管理→财务是审批和全局监控入口
- 这跟工时的多入口思路一致——干活的人在最顺手的地方操作，管钱的人在全局视角审批

---

## 三、预算成本管理作为独立对象线

**我说的：** 预算成本管理是独立模块，项目中心留个入口按钮进去。人工成本只是其中一个费用类别。

**设计产出：** 第9号线"预算成本管理"——3张表（budget_category + budget_detail + project_cost_record）+ 4个Tab界面（费用总览/人工成本/其他费用/费用记录）

**核心决策：**
- 人工成本不进工时模块，预算成本管理是独立模块
- 项目工时汇总页只留一个[💰预算成本管理]按钮，点进来就是
- 人工成本后端自动算（工时×时薪），前端不碰薪资原始数据
- 其他费用逐笔录入，预算先行再对比

## 四、9和10合并——拆两条线反而让人迷糊

**发现的问题：** 设计完第10号线"预算及执行"后，回头看第9号线"预算成本管理"，发现两件事本质是同一件事：

| | 第9号线 预算成本管理 | 第10号线 预算及执行 |
|---|---|---|
| 预算明细 | project_budget_detail | budget_item |
| 费用记录 | project_cost_record | — |
| 执行监控 | 预算对比Tab | budget_execution |
| 费用类别 | budget_category | 引用category_id |

**分析：** 9和10本质是同一件事：预算编制+审批+执行监控+费用归集，拆两条线反而让人迷糊。

**合并方案：9+10 → 一条"预算及执行"线**

删除的重复项：
- project_budget_detail → 被budget_item替代
- project_cost_record → 被cost_record替代
- 预算对比Tab → 就是budget_execution的展示

入口也简化了：
- 项目中心 → [💰预算及执行]：提请预算+看本项目执行
- 综合管理 → 财务 → 预算及执行：审批+全局监控
- 综合管理 → 财务 → 费用管理：报销/付款/借款/发票（第10号线）

**核心决策：财务线就两条线，边界清晰——管预算的（9+10合并）和花钱的（原11号线重编号为10号线）**

## 五、费用管理四合一

**我说的：** 好，费用报销+付款+发票，一条线做掉。

**设计产出：** 第10号线"费用管理"——报销+付款+借款+发票四合一

**为什么四合一：**
- 借款→报销→冲抵→发票是一条链，拆四个模块要跳来跳去
- 报销单里要关联发票，付款单里也要关联发票，发票单独管但随时能关联
- 借款最后是用报销来冲抵的，借款和报销必须在一起才能闭环
- 四合一不是堆功能，是因为业务就是一条链

**审批分级我定的：**
- ≤5K 经理→财务
- 5K~50K 加财务总监
- >50K 加CFO
- 这跟行业通用做法一致，CRO公司合同金额大，大额付款必须层层把关

## 六、三入口——跟工时一样的多入口思路

**发现的问题：** 费用管理最初只有综合管理一个入口，但项目PM想看本项目花了多少钱，中心想看本中心收了多少钱，都得跑到综合管理去筛，不方便。

**补三入口：**

- **综合管理 → 财务 → 费用管理**：全局视角，所有报销/付款/借款/发票
- **项目中心 → 某项目 → 费用Tab**：项目视角，只看本项目关联的报销+付款+预算执行，可直接新建（project_id自动带入）
- **中心驾驶舱 → 某中心 → 费用Tab**：中心视角，看本中心收款（管理费/检查费等）和关联报销，只读+发起视角

**核心决策：三入口数据同源，任一入口操作实时同步，跟工时的多入口设计一致**

项目中心入口：可看可建，project_id自动带入
中心驾驶舱入口：只读+发起，审批操作回到综合管理
综合管理入口：全局+审批

## 七、预算强控——花钱不能超预算

**联动规则：**
1. 提交报销/付款时自动校验关联预算，超预算80%预警，超预算100%阻断（需CFO特批）
2. 审批通过后自动写入第9号线cost_record(AUTO_COST)，更新budget_execution快照
3. 这样形成完整链路：预算审批→锁定→花钱→审批→记账→对比预算

**为什么超100%不是直接拦死而是要CFO特批：** CRO项目有时候必须花（比如受试者补偿不能停），完全拦死业务跑不了，但必须让CFO知道、签字。

---

## 八、项目预算不拆中心预算

**我定的：** 项目预算到项目级，不拆中心预算；中心预算后续需要时再补一层

**为什么：** 第一版先做项目级预算就够了，拆到中心级别数据量翻好几倍，而且中心费用大部分是从项目付款自动归集的，不需要单独做预算编制。等业务跑起来真需要了，加个budget_type=SITE就行。

---

## 九、人工成本自动算但不展示时薪

**计算规则：**
- 后端从timesheet.db读工时 × member.db读薪资
- 公式：hourly_rate = (base_salary + 社保公司部分 + 公积金公司部分) / 21.75 / 8
- 项目人工成本 = Σ(hourly_rate × actual_hours)
- 每月自动生成cost_record(source_type=AUTO_LABOR)

**但个人时薪不展示：** 前端只看汇总数据（按人员汇总金额、按岗位汇总金额），时薪仅后端计算用。这是隐私保护，薪资敏感。

## 十、整体数据流

```
7号线(人力资源) → 提供：人员、薪资、团队分配
       ↓
8号线(工时) → 提供：实际工时
       ↓
9号线(预算及执行) → 算：人工成本=工时×时薪 → 写cost_record(AUTO_LABOR) → 更新budget_execution
       ↑                    ↑
10号线(费用管理) → 花钱：报销/付款审批通过 → 写cost_record(AUTO_COST) → 更新budget_execution
```

7号线提供人，8号线提供工时，9号线算人工成本，10号线花钱写回9号线。四条线不是四个独立模块，是一条业务流。
