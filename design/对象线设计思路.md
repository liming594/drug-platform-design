# 对象线设计思路

> 为什么这样设计，而不是那样设计。

---

## 一、对象线是什么，为什么叫"线"

一个业务域就是一条线，线内自洽，线间松耦合。7号线人力资源、8号线工时、9号线预算及执行、10/11号线费用报销——每条线有自己的独立数据库、自己的路由文件、自己的前端模块。线和线之间通过ID引用关联，不共享数据库，不跨库写。

为什么不用一张大表或一个库？因为CRO业务的每个域生命周期不一样。人员入职离职是HR的事，工时填报是项目运营的事，预算审批是财务的事，费用报销是行政的事。如果把所有表塞进一个库，改一个字段影响全系统，迁移一个模块要动所有代码。独立库零耦合，每条线可以独立开发、独立部署、独立出问题不影响其他线。

为什么叫"线"不叫"模块"？因为线有方向——数据从7号线（人）流向8号线（工时），再从8号线流向9号线（预算执行），9号线又反向控制10号线（费用报销）。线是有流向的业务链，不是静态的功能块。

## 二、三入口设计——同一套数据，三个视角

每条线都有三个入口：综合管理（全局）、项目中心（项目维度）、中心驾驶舱（中心维度）。

为什么做三个入口而不是一个？因为CRO公司里不同角色看数据的角度完全不同：

- **HR看全公司**：谁入职了、谁离职了、组织架构怎么调——这是综合管理入口
- **PM看自己的项目**：PD-001项目上有几个人、CRA负载多少、CRC有没有空——这是项目中心入口
- **CRA看自己负责的中心**：协和中心这个月工时多少、费用多少——这是中心驾驶舱入口

但这三个入口**读写同一套表**，筛选条件不同而已。项目中心就是`WHERE project_id = :projectId`，中心驾驶舱就是`WHERE project_site_id = :siteId`，综合管理不加筛选看全部。

为什么不按入口分表？因为数据只有一份才不会打架。如果项目中心有一套项目人员表、综合管理又有一套人员表，两个入口改同一个东西就会数据不一致。三入口数据同源——任一入口操作实时同步——这是铁律。

工时线把这个思路走得更远：综合管理可以填、项目中心也可以填、中心驾驶舱也可以填。员工从综合管理统一入口填也行，在项目里顺手填也行，数据都写同一张weekly_timesheet。入口不同，但表是同一张，字段完全一样，只是项目/中心自动带出不用手动选。

## 三、7号线 人力资源——为什么TeamAssignment不拆表

传统做法是一个项目一张ProjectMember表、一个中心一张CenterMember表。但CRO的现实是：一个人同时负责多个项目的多个中心。张三可能PD-001项目上占40%、PD-002项目上占30%、还留30%做非项目工作。如果拆表，张三在ProjectMember里出现两条、在CenterMember里出现三条，改一个人的状态要改四五张表。

所以用一张TeamAssignment统一对象。`assignment_type`区分项目/中心/SMO/供应商，业务ID驱动过滤规则。一个人在多个项目/中心 = 多条记录，allocation_pct之和≤100%。

**人员替换**也是TeamAssignment的核心场景。CRA离职或转岗很常见，原记录标REPLACED、新建记录给接手人，保留完整替换链。离职的时候可以逐条交接（不同中心交给不同人），也可以批量交接（5个中心都给一个人）。这个操作在离职员单里做，不在项目人员管理里做——因为离职是HR的事，不是项目运营的事。

**岗位×级别=三维职级体系**：CRA序列×高级=高级CRA，CRC序列×中级=中级CRC。不搞职级表单独建，用position_type和position_level两个字段组合就够了。薪资、薪级、档位不进这张表——敏感信息隔离，第一版不做。

## 四、8号线 工时——为什么统一入口而不是项目内填

如果员工要进每个项目分别填工时，负责3个项目的CRA要切3次，很容易漏填。所以**统一从综合管理入口填报**，一个页面把本周所有项目+非项目工时一起填了。

**一周一填**，不按天填。CRO项目按周管理节奏更自然：周一到周日一个周期，周四16:00前填报。计划=本周五~下周四，实际=上周五~本周四。计划+实际同行——一条记录同时存plan和actual，不用填两次，后续对比偏差也方便。

**publisher_id逻辑**：PM/领导安排的工时，publisher_id=领导的member_id；自己填的，publisher_id=本人。这个字段后面做统计的时候很有用——领导安排了多少工时vs员工自己填了多少，可以看出管理力度。

**饱和度分级**：🔴偏低(<32h) ⚠️适中(32-36h) ✅正常(36-40h) 🟫超出(>40h)。CRO公司最怕的是CRC/CRA负荷不均——有人80%有人120%，所以饱和度是工时线的核心仪表盘。

**工时预算**：project_budget表存项目×角色的预算工时，跟实际工时对比。这是9号线人工成本的数据源头——工时×时薪=人工成本。

## 五、9号线 预算及执行——谁干活谁提请，管钱的审批监控

**核心原则就一句话：谁干活谁提请，管钱的审批监控。**

项目预算从项目中心提请（PM最清楚项目要花多少钱），审批在综合管理→财务（财务管钱当然要审批）。中心驾驶舱只看不管——中心主任不需要审批预算，但要能看到自己中心花了多少钱。

**预算审批后锁定**——APPROVED不可编辑，要改必须走变更流程。为什么？因为预算一旦批了就是承诺，不能PM自己偷偷改个数字。变更全留痕：每次变更生成新版本，保留历史快照，旧版本永远在。

**执行数据三种来源**：
- AUTO_LABOR：工时×时薪，从8号线weekly_timesheet和7号线member_salary自动算
- AUTO_COST：10号线费用报销审批通过后自动同步
- MANUAL：直接在费用录入Tab手动录入

为什么分三种？因为CRO项目的费用来源确实就是这三条路。人工成本是大头但不能让员工自己填（敏感），必须后端从工时+薪资算；报销付款走审批流程，审批通过就是确认的费用，自动同步不用手动录；还有些零碎费用（伦理费、快递费）直接手动录就行。

**个人时薪不展示**——前端只看汇总金额，时薪仅后端计算用。薪资是敏感信息，不能让PM看到某个CRC时薪80块。

**项目预算到项目级，不拆中心预算**——一个项目一份预算，中心维度只看费用汇总。为什么？因为CRO项目的预算管理是以项目为单位的，合同金额、预算总额都是项目级别。中心费用是项目费用的子集，从cost_record按project_site_id筛选即可。后续如果确实需要中心级预算，加个budget_type=SITE就行，不影响现有结构。

## 六、10/11号线 费用报销——四合一为什么不分四个模块

报销、付款、借款、发票——传统OA系统这是四个独立模块。但在CRO场景下，它们是一条链：

1. 出差→借款
2. 回来→报销→冲抵借款
3. 报销要附发票
4. 供应商付款也要发票

如果拆四个模块，用户要跨模块操作：借款在借款模块借、报销在报销模块报、冲抵要切到借款模块去冲、发票在发票模块去关联。太碎了。四合一放一个费用管理入口，一个页面看到所有单据，操作不用跳转。

**预算强控**：提交报销/付款时自动校验预算，超80%预警，超100%阻断需CFO特批。为什么不是超100%直接拒绝？因为CRO项目有时确实需要超预算——比如受试者入组超预期，受试者补偿必然超预算。CFO特批就是给这种情况留的口子，但必须留痕。

**金额分级审批**：
- ≤5K：经理→财务（日常小钱，不用惊动大领导）
- 5K~50K：加财务总监（中等金额，多一个人把关）
- \>50K：加CFO→总经理（大额支出，必须到最高层）

这是从实际业务来的——CRO公司大部分差旅报销在5K以下，SMO/中心付款在5-50K之间，CRO外包费经常超50K。分级审批让小钱走得快、大钱卡得严。

**借款冲抵闭环**：借款→报销→冲抵，逾期自动提醒。为什么逾期这么重要？因为CRO行业出差频繁，借款量大，如果没有人催，借款会越积越多。每周提醒借款人和经理，直到冲抵清零。

## 七、跨线数据流——7→8→9，9反向控制10

```
7号线(人) ──member_id──→ 8号线(工时) ──actual_hours──→ 9号线(预算执行)
                              │                              │
                              │ hourly_rate                  │ 预算校验
                              │ (from member_salary)         │ (超80%预警/超100%阻断)
                              ↓                              ↓
                        人工成本(AUTO_LABOR)         10号线(费用报销)
                        cost_record写入              审批通过→cost_record(AUTO_COST)
                                                     →刷新budget_execution
```

7号线提供人和薪资，8号线提供工时，9号线用人工×时薪算人工成本。10号线花钱，审批通过后写回9号线的cost_record和budget_execution，形成闭环。

**跨库联动怎么做的？** finance.db独立，审批通过后跨库写budget.db的cost_record。看似违背"零耦合"原则，但这是业务逻辑决定的——费用审批通过就必须更新预算执行，不更新数据就不一致。这里用的是应用层跨库写（同一个Node进程可以开两个db连接），不是共享数据库。如果以后要拆成微服务，这个跨库写改成HTTP API调用就行。

## 八、统一规则——四条线共用的设计决策

| 规则 | 为什么 |
|------|--------|
| 独立库零耦合 | 每条线独立开发、独立部署、独立出问题不影响其他线 |
| 逻辑外键不建物理FK | SQLite不支持跨库外键，应用层校验更灵活 |
| 跨域关联走ID引用 | 不共享DB，查询时应用层JOIN |
| 三入口数据同源 | 任一入口操作实时同步，避免数据不一致 |
| 打回必填原因 | 否则申请人不知道为什么被打回，反复提交反复打回 |
| 所有POST/PUT返回完整对象 | 前端不用二次查询，减少请求数 |
| 列表接口支持分页+排序 | 数据量大了不分页会卡 |
| 错误统一格式 | 前端统一处理，不用每个接口写不同的错误解析 |
| 审批后锁定 | 审批就是承诺，不能偷偷改 |
| 变更全留痕 | 审计需要，出了问题能追溯 |
| 自动编号(BX/FK/JK/FP) | 人眼可读，不用UUID做单据号 |

## 九、第一版不做什么——克制比功能更重要

每条线的文档最后都有"第一版不做"清单。不是想不到，是故意不做：

- **HR不做薪资管理**——敏感信息，第一版用现有HR系统/Excel维护，后面再接
- **工时不做审批流**——先自填自报跑起来，审批流后面加
- **工时不做移动端**——PC端先够用，移动端体验是另一个量级
- **预算不做中心级预算**——项目级预算够用，中心维度看费用汇总就行
- **预算不做自动排程**——预算执行快照每日刷新，不需要实时计算
- **费用不做发票OCR**——手动录入发票信息就行，OCR是锦上添花
- **不做仪表盘自定义拖拽**——固定5个仪表盘先跑，自定义是第二版的事

为什么克制？因为CRO项目是"先跑起来再迭代"。功能做多了交付慢、Bug多、用户还不会用。先把核心流程跑通，用户用起来提反馈，再按需加功能。
